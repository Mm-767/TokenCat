# 🐱 TokenCat

<p align="center">
  <img src="assets/tokencat-run.gif" width="252" alt="TokenCat — 달리는 픽셀 고양이">
</p>

macOS 메뉴바에서 픽셀 고양이가 뛰어다니고, Claude(Claude Code) 토큰 소모 속도가
빨라질수록 고양이도 빨라집니다. 클릭하면 5시간 세션·주간 사용량 팝오버가 열립니다.

- 😴 잠자기 → 🚶 산책 → 🏃 달리기 → 💨 질주 → 🌈 무지개 모드 (burn rate 기준 5단계)
- 사용률 80% 이상이면 🥵 지친 고양이, 95% 이상이면 ⚠️ 빨간 경고로 바뀝니다
- 80% / 95% 도달 시 macOS 알림 (창별 각 1회)
- 세션/주간 게이지는 계정 단위 공식 % (웹·데스크톱 사용량 포함), 고양이 속도는
  로컬 로그 기반 실시간

## 설치 (터미널 3줄)

```bash
git clone https://github.com/Mm-767/TokenCat.git
cd TokenCat
./install.sh
```

빌드 → 서명 → 응용 프로그램 폴더 설치 → 실행까지 한 번에 됩니다.
**요구사항**: macOS 13+, Claude Code 로그인 상태, Swift 툴체인
(없으면 `xcode-select --install` — Xcode 전체 설치는 필요 없습니다).
로컬에서 직접 빌드하므로 Gatekeeper "확인되지 않은 개발자" 차단이 없습니다.

### 첫 실행 시 (각 1회)

- **키체인 프롬프트**: "security가 키체인의 정보를 사용하려고 합니다"가 뜨면
  **[항상 허용]** 을 누르세요. [허용]만 누르면 다음에 다시 묻습니다.
- **서명 identity 생성**: 설치 중 macOS가 로그인 암호를 한 번 확인합니다
  (self-signed 인증서 신뢰 등록 — 정상 절차입니다).
- **알림 권한**: 한도 80%/95% 경고에 쓰입니다. 허용 권장.

### 껐다가 다시 켜기

메뉴바 전용 앱이라 Dock에는 뜨지 않습니다. 팝오버의 ⏻ 버튼으로 끈 뒤 다시 켜려면:

- **Spotlight**: `⌘ + Space` → "TokenCat" 입력 → Enter (제일 빠름)
- Launchpad 또는 응용 프로그램 폴더에서 TokenCat 실행
- 터미널: `open -a TokenCat`

매번 켜기 귀찮으면 **고양이 클릭 → ⚙️ 설정 → 일반 → "로그인 시 자동 시작"** 을
켜두세요. 재부팅해도 알아서 올라옵니다.

업데이트는 `git pull && ./install.sh`, 제거는 응용 프로그램 폴더에서
TokenCat.app 삭제.

## 개발

```bash
swift run TokenCat            # 개발 실행 (알림·자동 시작은 번들 앱에서만 동작)
swift test                    # 단위 테스트
.build/debug/TokenCat --report  # 집계 디버그 (ccusage 대조용)
scripts/build-app.sh          # 앱 번들 + dmg만 생성
```

ad-hoc 서명은 사용하지 않습니다 — 빌드마다 서명이 바뀌면 키체인·알림 권한
승인이 리셋되어 프롬프트가 반복되기 때문입니다 (docs/usage-endpoint.md의
"토큰 로드 설계" 참조). `scripts/setup-signing.sh`가 만드는 고정 identity
("TokenCat Dev")로 서명합니다.

## 데이터 소스 (하이브리드)

| 데이터 | 소스 |
|---|---|
| 세션/주간 게이지 % | **공식 OAuth usage 엔드포인트** — Claude Code가 로컬(macOS 키체인)에 저장한 토큰으로 조회. 웹/데스크톱 사용량까지 합산된 계정 단위 값이라 `/usage`와 일치. 180초 폴링 + 팝오버 열 때 재조회, 그 사이엔 로컬 소모분으로 보간("보간 중" 표기) |
| 고양이 속도·오늘 토큰·스파크라인 | `~/.claude/projects/**/*.jsonl` 로컬 파싱 (3초 폴링, 증분) |
| 폴백 (추정 모드) | 공식 조회 실패·off 시 플랜 프리셋 대비 JSONL 집계로 추정 — 모든 추정값에 "(추정)" 라벨 |

자세한 실측 스키마: [docs/jsonl-schema.md](docs/jsonl-schema.md), [docs/usage-endpoint.md](docs/usage-endpoint.md)

## 한계 (정직하게)

- 공식 % 조회는 **비문서화 API**입니다. 변경·차단될 수 있으며, 그 경우 자동으로
  추정 모드로 폴백합니다 (앱은 죽지 않습니다).
- 추정 모드의 플랜 한도는 커뮤니티 추정치입니다. 설정의 캘리브레이션(`/usage`
  실측 % 입력)으로 보정하세요.
- 2026-06-15부터 프로그래매틱 사용(Agent SDK, `claude -p`)은 별도 크레딧 풀입니다.
  JSONL 집계는 합산하되, entrypoint에 "sdk"가 포함된 레코드가 있으면 오늘 섹션에
  "프로그래매틱 N tokens 포함" 캡션으로 분리 표시합니다 (관용 판별 — 실측 데이터에
  SDK 레코드가 없어 마커는 추정).
- OAuth 토큰은 읽기 전용으로 사용하며 Anthropic 외 어디에도 전송·저장·로깅하지 않습니다.
- 키체인 읽기는 `/usr/bin/security` 서브프로세스 방식입니다. 키체인 프롬프트에서
  "항상 허용"을 누르면 이후 **다른 프로세스도 security CLI로 이 항목을 조용히 읽을 수
  있게 되는** 트레이드오프가 있습니다 (앱 리빌드 시 프롬프트 재발 방지와 맞바꾼 것).

## 🔧 트러블슈팅 기록

> 개발 과정에서 만난 굵직한 문제 2건을 원인 분석 → 설계 → 검증 순으로 기록한다.

### 사례 1. macOS 키체인 암호 프롬프트 반복 문제

**키워드**: macOS Keychain ACL · 코드서명 · OAuth 토큰 관리 · 폴링 설계 · 보안-편의 트레이드오프

**요약 (TL;DR)** — 메뉴바 앱이 Claude Code의 OAuth 토큰을 읽을 때마다 macOS 키체인
암호 프롬프트가 반복 발생했다. 원인을 "버그"가 아닌 **macOS 키체인 ACL의 정상 보안
동작**으로 규정하고, 접근 빈도·승인 방식·코드서명 3개 축으로 분해해 해결했다.
결과적으로 암호 프롬프트를 **최초 1회**로 줄이면서도 키체인 암호 저장 같은 보안
안티패턴은 배제했다.

#### 배경

TokenCat은 사용량 게이지의 정확도를 위해, Claude Code가 로컬에 저장한 OAuth 토큰으로
Anthropic 사용량 엔드포인트를 폴링(180초)한다. macOS에서 이 토큰은 키체인 항목
`Claude Code-credentials`(로그인 키체인)에 저장되어 있어, 앱이 이를 읽어야 했다.

#### 문제 현상

- 앱 실행 중 "키체인 항목에 접근하려고 합니다 — 암호를 입력하세요" 프롬프트가 계속 발생.
- "허용"을 눌러도 몇 분 뒤 다시 등장, 개발 중에는 빌드할 때마다 재발.
- 상시 실행 유틸리티에서 암호를 반복 입력해야 함 → 제품으로서 치명적 UX.

#### 원인 분석

먼저 "이 프롬프트는 왜 뜨는가"를 확인했다. macOS 키체인의 각 항목에는
**ACL(Access Control List)** 이 있어, 항목을 생성한 앱(Claude Code) 외의 프로세스가
비밀값을 읽으려 하면 시스템이 사용자 승인을 요구한다. 즉 **프롬프트 자체는 버그가
아니라 설계된 보안 동작**이다. 문제는 "왜 반복되는가"였고, 원인을 3가지로 분해했다.

| # | 원인 | 메커니즘 |
|---|---|---|
| 1 | 폴링마다 키체인 접근 | 180초 주기 폴링 코드가 매번 키체인을 읽음 → 승인 안 된 상태면 매번 프롬프트 |
| 2 | "허용"은 1회성 승인 | 프롬프트의 "허용(Allow)"은 해당 1회만 유효. ACL에 앱을 등록하는 것은 "항상 허용(Always Allow)" |
| 3 | 빌드마다 코드서명 변경 | 디버그 빌드를 ad-hoc 서명으로 반복 생성 → macOS는 서명이 바뀐 바이너리를 **다른 앱**으로 취급 → 기존 "항상 허용" 등록이 무효화되어 재프롬프트 |

특히 3번이 핵심이었다. "항상 허용을 눌렀는데도 또 뜬다"는 현상은 ACL이 앱의
**코드서명(designated requirement) 단위**로 신뢰를 기억하기 때문이었다.

#### 해결

원칙을 먼저 세웠다: **OS 보안 모델을 우회하지 않고, 그 규칙 안에서 접근 횟수와
신뢰 단위를 설계한다.**

**(1) 토큰 메모리 캐시 — 접근 빈도 자체를 줄임**
액세스 토큰 유효기간(~60분) 동안 메모리에 캐시하고, 만료 시에만 키체인을 재조회하도록
변경. 키체인 접근이 3분당 1회 → 시간당 1회로 감소 (약 1/20).

**(2) `/usr/bin/security` 서브프로세스 경유 — 신뢰 단위를 안정된 바이너리로 이동**
앱에서 Security.framework를 직접 호출하는 대신, Apple이 서명한 시스템 바이너리를
서브프로세스로 실행:

```bash
security find-generic-password -s "Claude Code-credentials" -a "$USER" -w
```

이렇게 하면 ACL의 "항상 허용" 등록 대상이 (빌드마다 바뀌는) 내 앱이 아니라 서명이
불변인 `/usr/bin/security`가 되어, **앱을 아무리 리빌드해도 프롬프트가 재발하지
않는다.**
*트레이드오프*: 이후 다른 프로세스도 security CLI로 해당 항목을 조용히 읽을 수 있게
됨 → README에 명시적으로 문서화하고 사용자가 선택하게 함.

**(3) 파일 폴백 체인 — 프롬프트 없는 경로 우선**
`~/.claude/.credentials.json`(파일 저장 환경)이 존재하면 키체인보다 먼저 사용.
최종 폴백 순서: **파일 → 키체인 → (실패 시) 추정 모드**로 기능 저하
운영(graceful degradation).

**(4) 고정 서명 identity — 개발 중 재프롬프트 차단**
디버그 빌드도 항상 동일한 서명 identity로 codesign (개발자 인증서 부재 시
self-signed 인증서를 생성·고정하는 `scripts/setup-signing.sh` 제공).
"항상 허용"이 빌드를 넘어 유지됨.

**하지 않은 것 (의도적 배제)** — `security unlock-keychain` 자동화, 키체인 암호
저장·자동입력. 프롬프트는 없앨 수 있지만 사용자 키체인 전체의 보안을 무너뜨리는
안티패턴이라 배제했다.

#### 결과

- 암호 프롬프트: 매 폴링/매 빌드 → **최초 1회("항상 허용") 이후 0회**.
- 폴링 안정성 유지: 사용량 게이지 기능 변화 없음, 실패 시 추정 모드로 자동 전환.
- 검증: 수정 후 반복 리빌드·재실행 중 프롬프트 재발 0회, 401 응답 시 캐시 무효화 →
  재읽기 → 1회 재시도 경로 동작 확인.

#### 배운 점

- **"버그처럼 보이는 보안 동작"을 구분하는 것이 첫 단계였다.** 증상을 없애려 하기
  전에, 시스템이 왜 그렇게 설계됐는지부터 확인하면 우회가 아닌 설계로 풀 수 있다.
- macOS 키체인 ACL은 코드서명 단위로 신뢰를 기억한다 — 데스크톱 앱에서 **서명 전략은
  배포뿐 아니라 런타임 UX 문제**다.
- 보안-편의 트레이드오프는 없앨 수 없다면 **문서화해서 사용자에게 선택권을 주는 것**이
  정직한 설계다.

### 사례 2. 실시간 보간 게이지 폭주 — 오탐 "100% 경고"

**키워드**: 이종 데이터 소스 결합 · 단위 불일치 · 실측 캘리브레이션 · 오탐 알림 방지

**요약 (TL;DR)** — 공식 사용률(%) 위에 로컬 로그 소모분을 얹는 실시간 보간 게이지가
몇 분 만에 100%로 폭주하며 오탐 경고를 냈다. 원인은 **단위가 다른 두 데이터를 고정
상수로 환산**한 것: 로컬 로그의 토큰 수는 캐시 읽기가 지배해 실제 한도 소진의 수십 배
스케일이었다. 환산 비율을 상수 대신 **실측 역산**(조회 시점 창 토큰 ÷ 공식 %)으로
바꿔 해결했다.

#### 배경

TokenCat의 게이지는 하이브리드다: 정확한 **공식 사용률 %**는 180초마다 폴링하고,
그 사이 공백은 3초마다 파싱하는 **로컬 JSONL 로그**의 소모분을 %로 환산해 얹어
실시간처럼 보여준다(보간).

#### 문제 현상

- 고양이가 잘 뛰다가 갑자기 게이지가 100%로 치솟으며 빨간 경고 상태로 전환.
- 도달할 리 없는 시점에 ⚠️ "한도 95%" 알림 발생 (오탐).
- 팝오버를 클릭해 열면 정상 %로 복귀 → 사용자에겐 "**눌러야 갱신되는 버그**"처럼 보임.
  (실제로는 클릭 시 공식 % 재조회로 보간이 리셋됐다가, 다음 180초 사이에 다시 폭주)

#### 원인 분석

보간분을 `이후 소모 토큰 ÷ 플랜 추정 한도(Pro 기준 500K)`로 계산하고 있었다.
그런데 로컬 로그의 토큰 수는 **프롬프트 캐시 읽기가 대부분**이라, 계정 한도가
소진되는 속도와는 완전히 다른 스케일이다.

| | 코드의 가정 | 실측 |
|---|---|---|
| 공식 1%에 해당하는 로컬 토큰 | 5,000 tokens (500K÷100) | **약 60,000 tokens** (블록 4M ÷ 공식 63%) |
| 5분간 30만 토큰 소모 시 보간 | **+60%p** (폭주) | +5.1%p |

즉 환산 비율이 **약 12배** 틀려 있었고, 폴링 간격(180초) 안에 게이지가 100%까지
치솟기에 충분했다. 잘못된 %는 게이지 표시로 끝나지 않고 고양이 상태 오버라이드(빨간
경고)와 임계 알림까지 오염시켰다 — **파생 값들이 같은 잘못된 스케일을 공유**하고
있었기 때문이다.

#### 해결

**(1) 실측 역산 캘리브레이션** — 환산 비율을 고정 상수가 아니라 매 공식 조회 시점에
역산: `1%당 토큰 수 = 조회 시점 창 토큰 ÷ 공식 %`. 시스템이 이미 주는 데이터(공식 %,
로컬 토큰)만으로 스케일이 자동 보정된다.

**(2) 불안정 구간 보간 생략** — 공식 %가 5% 미만(리셋 직후)이거나 창 토큰이 없으면
역산이 불안정하므로 보간을 생략하고 공식 값만 표시. 폭주 가능성을 구조적으로 차단.

**(3) 파생 값 스케일 일원화** — 알림의 "약 N분 분량 남음(현재 속도 기준)" 계산도
같은 역산 스케일로 교체(기존엔 잔여 시간이 수십 배 과소 표시). 엔진(고양이
상태·알림)과 팝오버(게이지)가 동일한 `GaugeMath` 모듈을 공유하도록 리팩터링 —
같은 공식이 두 곳에 중복 구현되며 어긋나는 것을 방지.

#### 결과

- 실측 시나리오(공식 63%, 블록 4M, 이후 30만 소모): 기존 123%→100% 폭주 → **68.1%**.
- 오탐 빨간 고양이·95% 알림 제거. 게이지는 180초마다 공식값에 수렴하고 그 사이엔
  현실적인 폭으로만 증가.
- `GaugeMath` 단위 테스트 6개 추가 (경계: 낮은 base, 데이터 부족, 100% 클램프,
  잔여 토큰 역산) — 총 53개 통과.

#### 배운 점

- **이종 데이터 소스를 섞을 때는 단위(스케일)부터 검증해야 한다.** "토큰"이라는 같은
  이름이라도 계정 한도의 토큰과 로그의 토큰은 다른 물리량이었다.
- 알 수 없는 환산 비율은 상수로 가정하지 말고 **시스템이 주는 실측값으로
  캘리브레이션**하는 편이 유지보수 없이 정확하다.
- 잘못된 값 하나가 게이지·상태·알림으로 전파됐다 — **파생 값들은 계산을 한 모듈로
  모아야** 오류도 한 번에 고쳐진다.
